# Code of Conduct

The short version: **be excellent to each other.** Treat the project,
the contributors, and the users with the same care you'd want when
you're the one filing the issue at 11pm before a release cut.

The long version is the [Contributor Covenant
2.1](https://www.contributor-covenant.org/version/2/1/code_of_conduct/),
which we adopt verbatim. It covers the standards you'd expect:
respectful communication, no harassment, no discrimination, focus on
the substance of contributions over personal characteristics.

## Scope

This Code of Conduct applies wherever Proxilion-the-project meets
people: this repository's issues / PRs / discussions, the
`hi@claygood.com` mailbox (the SECURITY.md disclosure address),
project-affiliated chats or events, and any place a contributor is
identifiably representing Proxilion.

It does *not* apply to disagreements about technical direction so
long as those disagreements are pursued in good faith. "Your
architecture is wrong because…" is a productive technical disagreement;
"…and you're an idiot for proposing it" crosses the line. Be specific,
cite the spec or the code, and assume the other party is also trying
to ship something good.

## Reporting

If something happens that you think violates this Code, mail
`hi@claygood.com` with `[proxilion-conduct]` in the subject. Reports
go to the project maintainer; we'll acknowledge within 72 hours and
keep the report confidential to the extent compatible with the
resolution. We won't share reporter identity with the subject of a
report without explicit consent.

For *security* vulnerabilities, use the separate channel documented
in [SECURITY.md](SECURITY.md) — `hi@claygood.com` with subject prefix
`[proxilion-sec]`. Conduct and security reports share the inbox but
not the response path.

## Enforcement

Possible responses, scaled to severity and pattern:

1. **Private clarification.** The first response to a minor lapse is
   usually a private message explaining what was off and what we'd
   like instead. Most reports never escalate past this.
2. **Public correction.** When a public exchange has set a bad tone,
   we may reply publicly to reset expectations without naming the
   reporter.
3. **Temporary restriction.** Time-boxed restriction from issues /
   PRs / discussions for repeated or more serious behavior.
4. **Permanent ban.** Reserved for severe behavior or sustained
   patterns that don't improve with the prior steps.

The maintainer's decision is final. There's no formal appeals
process — this is a small open-source project, not a corporate HR
function. If you think a decision was wrong, you're welcome to fork
the code under its MIT license and govern your fork however you
prefer.

## What this isn't

This document isn't a substitute for the actual work of running a
welcoming project. It's the floor, not the ceiling. The ceiling is
the body of contributors and the tone of the discussions — both of
which are everyone's job.
